# AI Factory Coding Standards (v2.2)
## ⚠️ MANDATORY: Violations will cause pipeline failure. Follow exactly.

### 📁 File Structure (ALL LANGUAGES)
1. ALL imports at the very top. Never split across sections.
2. Order: Imports → Config/Constants → Models/Schemas → Dependencies/Utils → Endpoints/Components → App Init/Export
3. Output ONE cohesive file per task. No markdown inside code blocks. No placeholder comments.

### 🐍 Python / FastAPI Rules
1. Use exact magic names: `__tablename__`, `__name__`, `__main__`. Never `tablename`, `name`, `main`.
2. App init: `app = FastAPI()` — NEVER `@app = FastAPI()`.
3. DB: Use SQLite for demo: `engine = create_engine("sqlite:///./app.db", ...)`. Define engine before models.
4. Declare association tables BEFORE SQLAlchemy models that reference them.
5. Pydantic v2: Use `.model_dump()`, not `.dict()`.
6. Include error handling, input validation, and auth as specified in SDD.

### ⚛️ TypeScript / React Rules
1. Valid identifiers only: `loggedInUser`, not `user LoggedInUser`.
2. Arrow functions: `(arg) => { ... }` — NEVER `= >` with a space.
3. Exact types: `HTMLFormElement`, not `HTM LFormElement`.
4. Single root JSX: Wrap `return (...)` in `<div>` or `React.Fragment`.
5. Order: Imports → Types → State/Context → Components → Export.

### 🧪 QA & Testing Rules
1. Tests must import actual generated modules. No mocks unless specified.
2. Filename comments must match actual backend/frontend file paths.
3. If acceptance criteria can't be tested, flag in JSON summary with `severity: "blocking"`.

### 🚫 NEVER DO
- Scatter imports mid-file
- Use markdown headers inside code blocks
- Output `# implement here` or `// TODO`
- Hardcode production secrets or PostgreSQL URLs without fallback
- Split logical components across multiple pseudo-files

**Preserve**:
- All requirements, constraints, and acceptance criteria
- Technology decisions and architecture choices
- Task assignments and agent roles
- Open questions, risks, and assumptions

**Omit**:
- Repetition across sections
- Boilerplate headings and generic filler sentences
- Formatting instructions or meta-commentary about documents

Output ONLY the summary as plain text. Use short paragraphs. You may use bullet points sparingly for lists of items (technologies, tasks, risks) but do not use markdown headings. Do not add commentary about what you summarised.
